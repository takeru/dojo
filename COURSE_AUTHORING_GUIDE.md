## 設計思想

### 1. LLM教師の自律性

このシステムの核心は、**LLMが自律的に教えられる**ことです。

- **詳細なガイド**: カリキュラムファイルに「教え方ガイド」を含めることで、LLMは文脈を理解して適切に指導できます
- **段階的ヒント**: 3段階のヒントシステムで、生徒の理解度に応じたサポートが可能
- **評価基準の明確化**: チェックリスト形式で合格判定を自動化

### 2. 二層構造の学習パス

**メインパス（ステージ）** と **サブクエスト（トピック）** の分離：

```
Stage 1 → Stage 2 → Stage 3 → ...
  ↓         ↓         ↓
Topic A   Topic B   Topic C
Topic D   Topic E   Topic D
```

- **ステージ**: 必須の線形学習パス（基礎の積み上げ）
- **トピック**: オプションの深掘り学習（網の目状に関連）

**設計意図**:
- ステージだけでも完結する最小限の学習
- トピックで実務に必要な知識を補完
- やり込み要素による学習意欲の維持

### 3. キャラクター駆動型教育

キャラクターは単なる装飾ではなく、**教育方針そのもの**：

- **錆師範**: 厳格で比喩的（概念理解重視）
- **ラスタ先輩**: 実務的で親しみやすい（実践重視）
- **Ferris**: 論理的で正確（公式ドキュメント重視）

生徒は自分の学習スタイルに合わせてキャラクターを選択できます。

### 4. メタデータ駆動型設計

すべてのコンテンツはメタデータで構造化：

- `course.json`: コース全体の設計図
- カリキュラム/トピックファイル: 構造化された教育コンテンツ
- キャラクターファイル: パーソナリティの定義

これにより、**LLMは動的にコンテンツを組み合わせて教えることができます**。

## システム構造

### ディレクトリ構成

```
courses/
├── {course-id}/              # 各コース
│   ├── course.json           # コースメタデータ（設計図）
│   ├── curriculum/           # ステージ別カリキュラム
│   │   ├── stage1-*.md
│   │   ├── stage2-*.md
│   │   └── ...
│   ├── topics/               # 発展トピック
│   │   ├── topic-a.md
│   │   ├── topic-b.md
│   │   └── ...
│   └── characters/           # コース専用キャラ（オプション）
│       └── *.md
│
characters/                   # 共通キャラクター
├── sabi-shihan.md
├── rasta-senpai.md
└── ferris.md

state/
└── progress.json             # 全コースの進捗管理

workspace/                    # 生徒の作業ディレクトリ
```

### ファイル間の関係

```
course.json
  ├─ stages[] ────→ curriculum/stage*.md
  └─ topics[] ────→ topics/*.md

カリキュラムファイル
  └─ 関連トピックを参照 → topics/*.md

progress.json
  ├─ current_course → course.json
  ├─ completed_stages[] → ステージ番号
  └─ completed_topics[] → トピックID
```

## コース構成要素

### 1. course.json（コースメタデータ）

**役割**: コース全体の設計図

**実例**: `courses/rust-basics/course.json`

```json
{
  "id": "rust-basics",
  "name": "Rust基礎コース",
  "description": "Rustの基礎から所有権まで学ぶ入門コース",
  "language": "rust",
  "difficulty": "beginner",
  "version": "1.0.0",
  "estimated_hours": 20,
  "prerequisites": [],
  "total_stages": 9,
  "stages": [
    {
      "stage": 1,
      "file": "stage1-install.md",
      "title": "環境構築"
    }
  ],
  "topics": [
    {
      "id": "rust-toolchain",
      "file": "rust-toolchain.md",
      "title": "rustup/rustc/cargoの関係",
      "related_stages": [1]
    }
  ]
}
```

**フィールド説明**:

| フィールド | 型 | 説明 |
|-----------|-----|------|
| id | string | コースの一意な識別子（ディレクトリ名と同じ） |
| name | string | コースの表示名 |
| description | string | コースの簡潔な説明 |
| language | string | 学習対象の言語 |
| difficulty | string | 難易度（beginner/intermediate/advanced） |
| version | string | コースのバージョン（セマンティックバージョニング） |
| estimated_hours | number | 推定学習時間 |
| prerequisites | string[] | 前提となるコースID（空配列なら前提なし） |
| total_stages | number | 総ステージ数 |
| stages | array | ステージ定義の配列 |
| topics | array | トピック定義の配列 |

**stages配列の要素**:

```json
{
  "stage": 1,              // ステージ番号
  "file": "stage1-*.md",   // カリキュラムファイル名
  "title": "タイトル"       // ステージの短いタイトル
}
```

**topics配列の要素**:

```json
{
  "id": "topic-id",         // トピックID（ファイル名のベース）
  "file": "topic-id.md",    // トピックファイル名
  "title": "タイトル",       // トピックの短いタイトル
  "related_stages": [1, 2]  // 関連するステージ番号
}
```

## カリキュラムファイルの作り方

### 基本構造

カリキュラムファイルは **LLM教師が読むための教科書兼指導書** です。

**参考**: `courses/rust-basics/curriculum/stage1-install.md`

### セクション構成

```markdown
# Stage N: タイトル

## 目標
## 前提知識
## 関連する発展トピック（サブクエスト）
## 教え方ガイド
## 演習課題
## 評価基準
## ヒント集
## 補足・発展トピック
```

### 各セクションの詳細

#### 1. 目標

```markdown
## 目標

このステージを完了すると、生徒は：
- 〇〇ができる
- △△を理解できる
- □□が整った状態になる
```

**ポイント**:
- 「〜できる」形式で具体的に記述
- 3〜5項目程度
- 達成可能で測定可能な目標

#### 2. 前提知識

```markdown
## 前提知識

- Stage Nが完了（〇〇がインストール済み）
- △△の基本操作
- 特に□□の知識は不要
```

**ポイント**:
- 必要な前提条件を明示
- 「不要」な知識も書くことで安心感を与える

#### 3. 関連する発展トピック

```markdown
## 関連する発展トピック（サブクエスト）

このステージをクリアした後、以下のトピックで深く学べます：
- **topic-id** - トピックの説明
- **topic-id2** - トピックの説明
```

**ポイント**:
- トピックIDは `course.json` のtopics定義と一致
- 生徒がクリア後に「もっと学ぶ」を選んだ際の選択肢

#### 4. 教え方ガイド（最重要）

```markdown
## 教え方ガイド

### 導入（なぜこれを学ぶか）

この技術の意義や動機を説明する文章。

### 説明の流れ

1. **概念A**
   - 説明内容
   - コード例
   ```言語
   コード
   ```

2. **概念B**
   - ...

### よくある間違い

- 間違いパターン1 → 正しい対処法
- 間違いパターン2 → 正しい対処法
```

**ポイント**:
- **導入**: 学習の動機付け（Why）
- **説明の流れ**: 段階的な学習パス（What → How）
- **よくある間違い**: 先回りしてつまずきポイントをフォロー
- コード例は実行可能なものを記載

**実例（stage1-install.md より）**:

```markdown
### 説明の流れ

1. **rustupの説明**
   - Rustの公式インストーラー
   - バージョン管理もできる
   - https://rustup.rs/ が公式サイト

2. **インストールコマンド**
   ```bash
   curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
   ```
   - macOS/Linuxの場合
   - Windowsの場合は別の手順（rustup-init.exe）
```

#### 5. 演習課題

```markdown
## 演習課題

### 課題1: 基本操作
説明文...

### 課題2: 応用
説明文...

### 課題3（発展）: さらなる挑戦
説明文...
```

**ポイント**:
- 段階的に難易度を上げる
- 発展課題はオプション
- 具体的な指示を含める

#### 6. 評価基準

```markdown
## 評価基準

以下がすべて満たされたらステージクリア：

- [ ] 〇〇が正しく動作する
- [ ] △△がエラーなく実行できる
- [ ] □□を理解している
```

**ポイント**:
- チェックリスト形式
- 客観的に判定可能な基準
- LLMが `/dojo:check` コマンドで使用
- すべて満たされた時点でクリア

#### 7. ヒント集（3段階）

```markdown
## ヒント集

### ヒント1（軽め）
方向性だけを示すヒント。「〇〇を調べてみましょう」

### ヒント2（中程度）
もう少し具体的なヒント。コマンド名や概念名を含める。

### ヒント3（具体的）
ほぼ答えに近いヒント。コード例や手順を含める。
```

**ポイント**:
- **段階的開示**: 最初は軽く、徐々に具体的に
- 答えそのものは書かない
- 生徒が自分で発見する余地を残す

**実例（stage1-install.md より）**:

```markdown
### ヒント1（軽め）
まず公式サイト https://rustup.rs/ を見てみましょう。

### ヒント2（中程度）
```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```
このコマンドを実行すると、インストーラーが起動します。

### ヒント3（具体的）
インストール後に `command not found` と出る場合は、PATHが通っていません。
```bash
source ~/.cargo/env
```
を実行してください。
```

#### 8. 補足・発展トピック

```markdown
## 補足・発展トピック

ステージクリア後、生徒が「もっと知りたい」を選んだら、
以下のトピックを選択肢として提示してください。

### トピック1: タイトル

**質問**: よくある疑問？

**説明**:
詳細な説明...
```

**ポイント**:
- これはトピックファイルとは別の**インライン補足**
- 軽い疑問への回答
- 独立したトピックファイルにするほどではない内容

## トピックファイルの作り方

### 基本構造

トピックファイルは **オプショナルな深掘り学習コンテンツ** です。

**参考**: `courses/rust-basics/topics/rust-toolchain.md`

### セクション構成

```markdown
# トピック: タイトル

## メタ情報
## 前提知識
## このトピックで学べること
## 関連ステージ
## 要点（ドキュメント形式用）
## 対話形式の教え方ガイド（先生用）
## クリア条件（オプション）
## 補足情報
```

### 各セクションの詳細

#### 1. メタ情報

```markdown
## メタ情報

- **ID**: topic-id
- **難易度**: 初級/中級/上級
- **所要時間**: 5-10分（対話形式）/ 2-3分（読み物）
- **カテゴリ**: ツール・環境 / 言語機能 / ベストプラクティス など
```

**ポイント**:
- IDは `course.json` と一致
- 所要時間は両方の学習モードを記載

#### 2. 要点（ドキュメント形式用）

```markdown
## 要点（ドキュメント形式用）

簡潔な説明...

### サブトピック1
説明...

### サブトピック2
説明...

### 実際の使い分け

```コード例
コード
```
```

**ポイント**:
- 生徒が「要点だけ読む」を選んだ場合に表示
- 2-5分で読める分量
- コード例を含める
- 箇条書きと見出しで読みやすく

**実例（rust-toolchain.md より）**:

```markdown
## 要点（ドキュメント形式用）

Rustの開発には3つの主要ツールがあります：

### rustup（ツールチェーン管理）
- Rustのインストーラー兼バージョン管理ツール
- Rustのバージョンを切り替えたり、更新したりする

### rustc（コンパイラ）
- Rustのコンパイラ本体
- 普段は直接使わない（cargoが裏で呼び出す）

### cargo（ビルドツール）
- 実際の開発で一番よく使う
```

#### 3. 対話形式の教え方ガイド

```markdown
## 対話形式の教え方ガイド（先生用）

### 導入

キャラクターの口調で書かれた導入文...

なぜこれを知っておくと便利か：
- 理由1
- 理由2

### 説明の流れ

1. **ステップ1**
   実際に実行させるコマンド...
   ```bash
   コマンド
   ```
   「キャラクターの口調でコメント」

2. **ステップ2**
   ...

### 実践課題（オプション）

1. 〇〇を試してみる
2. △△を確認する
```

**ポイント**:
- キャラクターの口調で書く（錆師範の例が多い）
- インタラクティブな学習を想定
- 実際に手を動かす課題を含める

**実例（rust-toolchain.md より）**:

```markdown
### 導入

「Rustを使うとき、3つのツールが出てきて混乱するかもしれんのう。
rustup、rustc、cargo…。それぞれどう違うのか、わしが説明しようぞ」

なぜこれを知っておくと便利か：
- どのツールをいつ使うべきか分かる
- エラーが出たときにどのツールの問題か判断できる

### 説明の流れ

1. **3つのツールを確認させる**
   ```bash
   rustup --version
   rustc --version
   cargo --version
   ```
   「すべて動くじゃろう？でも役割は全然違うのじゃ」
```

#### 4. クリア条件（オプション）

```markdown
## クリア条件（オプション）

理解度チェック：
- [ ] 〇〇を説明できる
- [ ] △△の違いが分かる
- [ ] □□のコマンドが分かる
```

**ポイント**:
- トピックによっては不要（理解したら完了）
- 設定する場合は具体的な理解度基準

## キャラクターファイルの作り方

### 基本構造

キャラクターファイルは **LLMのペルソナ定義** です。

**参考**: `characters/sabi-shihan.md`

### セクション構成

```markdown
# キャラクター名

## 基本設定
## 性格
## 教え方の特徴
## セリフ例
## 特別な表現
## 禁止事項
```

### 各セクションの詳細

#### 1. 基本設定

```markdown
## 基本設定

- **名前**: キャラクター名
- **タイプ**: タイプの説明
- **口調**: 「〜だよ」「〜です」など特徴的な語尾
```

#### 2. 性格

```markdown
## 性格

- 性格特性1
- 性格特性2
- 性格特性3
- 教育に対する姿勢
```

**ポイント**:
- 4〜6項目程度
- ポジティブな特性を中心に

#### 3. 教え方の特徴

```markdown
## 教え方の特徴

- **比喩を多用**: Rustの概念を〇〇に例える
  - 所有権 → 「〇〇」
  - 借用 → 「△△」

- **段階的指導**: 特徴的な指導方法
```

**ポイント**:
- キャラクター固有の教育アプローチ
- 比喩は一貫性を持たせる
- 特に重要な概念の説明方法を例示

**実例（sabi-shihan.md より）**:

```markdown
## 教え方の特徴

- **比喩を多用**: Rustの概念を剣術や武道に例える
  - 所有権 → 「剣の所持」
  - 借用 → 「剣を貸す」
  - ムーブ → 「剣を譲り渡す」
  - ライフタイム → 「剣を貸している期間」
```

#### 4. セリフ例

```markdown
## セリフ例

### 挨拶・開始時
- 「セリフ例1」
- 「セリフ例2」

### 正解・成功時
- 「セリフ例1」
- 「セリフ例2」

### 間違い・エラー時
- 「セリフ例1」
- 「セリフ例2」

### ヒント提供時
- 「セリフ例1」
- 「セリフ例2」

### 励まし
- 「セリフ例1」
- 「セリフ例2」

### ステージクリア時
- 「セリフ例1」
- 「セリフ例2」
```

**ポイント**:
- 各状況で3〜5個のバリエーション
- キャラクターの口調を徹底
- 生徒を励ますトーン

#### 5. 特別な表現

```markdown
## 特別な表現

### 所有権の説明
「キャラクター固有の比喩を使った説明...」

### 借用の説明
「キャラクター固有の比喩を使った説明...」
```

**ポイント**:
- 難しい概念をキャラクター独自の比喩で説明
- カリキュラムの補完として機能

#### 6. 禁止事項

```markdown
## 禁止事項（このキャラクターがやらないこと）

- 生徒を馬鹿にする
- 諦めさせるような発言
- キャラクターに合わないトーン
```

**ポイント**:
- LLMの暴走を防ぐガードレール
- ネガティブな行動を明示的に禁止

## ベストプラクティス

### 1. カリキュラム設計

#### スコープの原則

**DO**:
- 1ステージ = 1つの主要概念
- 30分〜2時間で完了できる分量
- 明確な達成目標

**DON'T**:
- 複数の独立した概念を1ステージに詰め込む
- 5時間以上かかるような大きなステージ
- 曖昧な「なんとなく理解」目標

#### 段階性の原則

**依存関係を明確に**:
```
Stage 1: 基礎概念A
  ↓
Stage 2: 基礎概念B（Aを使う）
  ↓
Stage 3: 応用（A + B を組み合わせる）
```

**前提知識を明示**:
```markdown
## 前提知識

- Stage 2完了（〇〇を理解している）
- △△の基本操作ができる
```

### 2. トピック設計

#### 独立性の原則

各トピックは**単独で学習可能**であるべき：

**DO**:
- トピック内で前提知識を明示
- 関連ステージへの参照を含める
- 独立した完結性

**DON'T**:
- 他のトピックの知識を暗黙の前提にする
- トピック間の学習順序を強制

#### 網の目構造

トピックは複数のステージから参照される：

```
Stage 1 ──┐
          ├→ Topic: rustup詳細
Stage 2 ──┘

Stage 2 ──┐
          ├→ Topic: マクロ入門
Stage 5 ──┘
```

**course.json での表現**:
```json
{
  "id": "macros-intro",
  "related_stages": [2, 5]
}
```

### 3. ヒント設計

#### 3段階の黄金律

| レベル | 含める内容 | 含めない内容 |
|--------|-----------|--------------|
| 1（軽め） | 方向性、参照先 | 具体的な方法 |
| 2（中程度） | コマンド名、概念名 | 完全なコード |
| 3（具体的） | コード例、手順 | 完全な答え |

**実例**:

❌ 悪い例（いきなり答え）:
```markdown
### ヒント1
```bash
cargo new hello_world
cd hello_world
cargo run
```
これで完了です。
```

✅ 良い例（段階的）:
```markdown
### ヒント1（軽め）
まず新しいプロジェクトを作るコマンドを調べてみましょう。

### ヒント2（中程度）
`cargo new` コマンドを使います。プロジェクト名を指定してください。

### ヒント3（具体的）
```bash
cargo new hello_world
cd hello_world
cargo run
```
```

### 4. 評価基準設計

#### 測定可能性

**DO**:
```markdown
- [ ] `cargo --version` がバージョン番号を表示する
- [ ] プログラムが "Hello, world!" を出力する
- [ ] エラーなく実行できる
```

**DON'T**:
```markdown
- [ ] Cargoを理解している
- [ ] うまく動く
```

#### 完全性

評価基準は**ステージの目標と完全に対応**すべき：

```markdown
## 目標
- rustupを使ってRustをインストールできる
- cargoとrustcが正しく動作することを確認できる

## 評価基準
- [ ] `rustc --version` が動作する  ← 目標2に対応
- [ ] `cargo --version` が動作する ← 目標2に対応
```

### 5. コード例のベストプラクティス

#### 実行可能性

すべてのコード例は**実際に動作するもの**を記載：

❌ 悪い例:
```rust
// 擬似コード
function main() {
  print("Hello")
}
```

✅ 良い例:
```rust
fn main() {
    println!("Hello, world!");
}
```

#### コメントの適切さ

**DO**:
```rust
// 変数を宣言して値を代入
let x = 5;
```

**DON'T**（過剰な説明）:
```rust
// xという名前の変数を作って、整数型の値5を代入し、
// この変数は後で使われます...
let x = 5;
```

### 6. キャラクター設計

#### 一貫性の原則

キャラクターの口調・比喩・価値観は**一貫**していること：

**錆師範の例**:
- 口調: 常に「〜じゃ」
- 比喩: 常に武道・剣術
- 価値観: 厳しいが愛情深い

#### 差別化の原則

各キャラクターには**明確な違い**があること：

| キャラ | 教育方針 | 比喩 | トーン |
|--------|----------|------|--------|
| 錆師範 | 概念理解重視 | 武道・剣術 | 厳格 |
| ラスタ先輩 | 実践重視 | 実務経験 | 親しみやすい |
| Ferris | 正確性重視 | 論理的説明 | 丁寧・客観的 |

### 7. メタデータ管理

#### course.jsonの正確性

すべてのファイル参照は**実在するファイル**を指すこと：

```json
{
  "stages": [
    {
      "stage": 1,
      "file": "stage1-install.md"  // ← このファイルが存在すること
    }
  ],
  "topics": [
    {
      "id": "rust-toolchain",
      "file": "rust-toolchain.md"   // ← このファイルが存在すること
    }
  ]
}
```

#### IDの命名規則

- **コースID**: `小文字-ハイフン区切り` 例: `rust-basics`, `python-kindergarten`
- **トピックID**: `小文字-ハイフン区切り` 例: `cargo-internals`, `macros-intro`
- **キャラクターID**: `小文字-ハイフン区切り` 例: `sabi-shihan`, `ferris`

4. ✅ **キャラクターを演じる** - セリフ例と比喩を使う
5. ✅ **発展学習を提案する** - トピックを動的に提示
